# Benchmarks

Methodology: cold `tsc --noEmit` (fresh Node process each run) vs one warm `Session` checking N in-memory overlays. Disk is never written by the warm path. Numbers below were measured on this machine (Fedora, Node 22) on 2026-07-08. Re-run before quoting.

## Reproduce

```bash
# Bundled 3-file fixture (default)
npm run bench

# Synthetic 200-module project (generated under bench/large/, gitignored)
npm run bench:large

# Any real repo
npm run bench -- --project path/to/tsconfig.json --runs 5 --file src/some.ts
```

Flags for `bench/bench.mjs`:

| Flag | Default | Meaning |
| --- | --- | --- |
| `--project` | `test/fixture/tsconfig.json` | tsconfig to load |
| `--runs` | `5` | cold and warm samples |
| `--file` | auto (baseline error file, else `src/entry.ts` / `src/report.ts`) | project-relative file to overlay |

The warm overlay appends a trailing comment so the file changes without inventing a type error. That measures incremental recheck cost, not "fix a bug" cost.

## Measured (this machine)

### Bundled fixture (`test/fixture`, 2 root files, 1 baseline error)

| what | cost |
| --- | --- |
| cold `tsc --noEmit`, per candidate | **1,018 ms** avg (n=5) |
| veredicto session init (incl. baseline) | **384 ms** once |
| veredicto warm check, per candidate | **8.5 ms** avg (n=5) |
| per-candidate speedup after init | **~119×** |

### Synthetic large project (`bench/large`, 201 root files, 1 baseline error)

Generated by `node bench/generate-large.mjs` (200 chained modules + `entry.ts`).

| what | cost |
| --- | --- |
| cold `tsc --noEmit`, per candidate | **424 ms** avg (n=5) |
| veredicto session init (incl. baseline) | **492 ms** once |
| veredicto warm check, per candidate | **110.3 ms** avg (n=5) |
| per-candidate speedup after init | **~4×** |

## How to read these numbers

- **Fixture ~119×** shows the product thesis on a tiny project: you stop paying process startup + full program construction per attempt.
- **Large ~4×** is the honest large-synthetic result on a warm machine. Cold `tsc` here benefits from OS page cache across the five sequential cold runs; a truly cold disk or a slower CI container widens the gap (the original README container measured ~6.6 s cold on the fixture). Init on 201 files already exceeds one cold run — that is expected; you pay init once per session.
- Warm cost scales with what the LanguageService must recheck after the overlay, not with "restart TypeScript from zero."
- **Do not quote the fixture 119× for a monorepo.** Run `npm run bench -- --project …` on the repo you care about and file the numbers.

## What this does *not* measure

- Unified-diff apply cost (v1 is full-file overlays)
- Parallel candidates (v1 is sequential)
- Network round-trip to `POST /v1/check` (add your own if the daemon is remote — it shouldn't be; loopback only)
- Token savings from `--compact` / structured JSON vs scraping `tsc` prose (qualitative; measure in your agent loop)
